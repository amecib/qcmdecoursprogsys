CH 01 🖥️ Introduction – OS & UNIX

Rôle de l'OS, architecture UNIX, mode noyau/utilisateur, historique

1
Quelles sont les deux fonctions principales d'un système d'exploitation ? 🔥 examen
  • A) Gérer le clavier et l'écran
  • B) Machine étendue (abstraction) + Gestionnaire de ressources
  • C) Compiler les programmes et les exécuter
  • D) Assurer la connexion réseau et stocker les fichiers
👁 Voir la correction
✅ Bonne réponse : B
Machine étendue : l'OS masque la complexité du matériel et fournit une interface simplifiée.
Gestionnaire de ressources : il contrôle et arbitre l'accès au CPU, à la mémoire, aux périphériques et aux données.

👉 Les 4 ressources à citer : CPU, mémoire, système de fichiers, entrées/sorties. Oublier l'une d'elles coûte des points.
2
Le shell UNIX fait-il partie du noyau ? 🔥 examen
  • A) Oui, le shell est intégré au noyau Linux
  • B) Non, le shell est un programme utilisateur qui utilise l'API système
  • C) Oui, mais uniquement bash
  • D) Non, le shell est un compilateur indépendant
👁 Voir la correction
✅ Bonne réponse : B
Non. Le shell est un programme utilisateur (couche application) qui utilise les appels système pour communiquer avec le noyau. Architecture UNIX en 3 couches :
① Matériel → ② Noyau (kernel) → ③ Shell & applications.

Seul le noyau interagit directement avec le matériel. Les interpréteurs et compilateurs ne font pas partie du noyau.
3
À quelle section du manuel appartiennent les appels système ? 🔥 examen
  • A) Section 1 (commandes shell)
  • B) Section 2 (appels système)
  • C) Section 3 (bibliothèques C)
  • D) Section 8 (administration)
👁 Voir la correction
✅ Bonne réponse : B
Sections du manuel Unix :
• 1 → Commandes shell (man printf → commande shell)
• 2 → Appels système (man 2 fork, man 2 open)
• 3 → Fonctions bibliothèque C (man 3 printf → fonction C)
• 8 → Commandes d'administration root

⚠️ Piège classique : man printf ≠ man 3 printf !
4
En mode noyau (kernel mode), le processeur peut :
  • A) Exécuter uniquement les calculs arithmétiques
  • B) Exécuter n'importe quelle instruction, y compris l'accès direct au matériel
  • C) Seulement lire la mémoire, pas y écrire
  • D) Exécuter les mêmes instructions qu'en mode utilisateur
👁 Voir la correction
✅ Bonne réponse : B
Le processeur a deux modes :
• Mode noyau (privilégié) : toutes les instructions sont autorisées, accès direct au matériel.
• Mode utilisateur : sous-ensemble limité d'instructions non critiques.

Une application passe en mode noyau via un appel système (interruption logicielle), exécute l'opération privilégiée, puis repasse en mode utilisateur.
5
Le concept de "temps partagé" (time sharing) repose sur :
  • A) Un seul programme en mémoire à la fois
  • B) Plusieurs utilisateurs avec des disques séparés
  • C) Des quanta de temps CPU alloués à chaque processus, donnant l'illusion de parallélisme
  • D) L'exécution parallèle réelle sur plusieurs CPU
👁 Voir la correction
✅ Bonne réponse : C
Le temps partagé découpe le CPU en quanta (intervalles très courts). Le CPU bascule rapidement entre les programmes, donnant l'illusion à chaque utilisateur qu'il est seul sur la machine.

Un seul CPU (1 cœur) n'exécute qu'un seul programme à la fois : c'est l'OS qui crée l'illusion de parallélisme.
6
UNIX a été créé par Ken Thompson et Dennis Ritchie en :
  • A) 1985
  • B) 1973 (avec la réécriture en C)
  • C) 1969 (chez Bell Labs, AT&T)
  • D) 1991 (par Linus Torvalds)
👁 Voir la correction
✅ Bonne réponse : C
• 1969 : création d'UNIX par Ken Thompson et Dennis Ritchie chez Bell Labs (AT&T).
• 1973 : réécriture d'UNIX en langage C (inventé par Dennis Ritchie).
• 1991 : Linux créé par Linus Torvalds — clone libre d'UNIX.

Linux ≠ UNIX, mais Linux est compatible UNIX (POSIX).
CH 02 📁 Système de fichiers UNIX

Arborescence, permissions rwx, chmod, chemins, grep, sed, find

7
Que signifie chmod 755 script.sh ? 🔥 examen
  • A) rwxrwxrwx
  • B) rw-r--r--
  • C) rwxr-xr-x
  • D) rwxrwxr-x
👁 Voir la correction
✅ Bonne réponse : C — rwxr-xr-x
Calcul : r=4, w=2, x=1
• 7 = 4+2+1 = rwx (propriétaire : lecture + écriture + exécution)
• 5 = 4+1 = r-x (groupe : lecture + exécution)
• 5 = 4+1 = r-x (autres : lecture + exécution)

Mémo rapide :
• chmod 755 → rwxr-xr-x (script standard)
• chmod 644 → rw-r--r-- (fichier texte)
• chmod 750 → rwxr-x--- (script groupe seulement)

⚠️ Piège : chmod 75 (2 chiffres) est invalide. Toujours 3 chiffres.
8
Interpréter cette ligne : drwxr-x--- 3 root staff 4096 10 mai data 🔥 examen
  • A) Fichier ordinaire, root peut tout faire, staff peut lire seulement
  • B) Répertoire, root peut tout faire, staff peut lire et traverser, les autres : rien
  • C) Lien symbolique, permissions complètes pour tous
  • D) Répertoire, staff peut écrire, les autres peuvent lire
👁 Voir la correction
✅ Bonne réponse : B
Décodage complet :
• d → répertoire (d=directory, -=fichier, l=lien)
• rwx → propriétaire (root) : lecture + écriture + traversée
• r-x → groupe (staff) : lecture + traversée (pas d'écriture)
• --- → autres : aucun droit
• 3 → 3 liens physiques
• 4096 → taille en octets
• data → nom du répertoire

⚠️ Pour un répertoire, x = permission d'entrer (cd), pas d'exécuter.
9
Quel droit est nécessaire pour entrer dans un répertoire avec cd ? ⚠️ piège
  • A) Lecture (r)
  • B) Écriture (w)
  • C) Exécution (x)
  • D) Lecture ET exécution (r+x)
👁 Voir la correction
✅ Bonne réponse : C — Exécution (x) seule
Pour un répertoire, les droits ont une signification différente des fichiers :
• r → lire la liste des fichiers (ls)
• w → créer/supprimer des fichiers dans le répertoire
• x → traverser / entrer dans le répertoire (cd)

⚠️ Piège classique : beaucoup répondent "r" en pensant "voir le contenu" mais pour entrer, il faut x.
10
Un chemin absolu se distingue d'un chemin relatif parce qu'il :
  • A) Commence par . ou ..
  • B) Commence toujours par /
  • C) Contient le nom de l'utilisateur
  • D) Ne contient jamais de /
👁 Voir la correction
✅ Bonne réponse : B
• Chemin absolu : part de la racine /. Commence toujours par /. Ex : /home/user/tp1/main.c
• Chemin relatif : part du répertoire courant. Ne commence jamais par /. Ex : ../tp1/main.c

Symboles spéciaux : . = répertoire courant, .. = parent, ~ = home.
11
Que fait grep -v "^#" fichier.c | wc -l ? 🔥 examen
  • A) Compte les lignes commençant par #
  • B) Compte les lignes ne commençant pas par # (lignes actives)
  • C) Supprime les commentaires du fichier
  • D) Liste les fichiers .c contenant des commentaires
👁 Voir la correction
✅ Bonne réponse : B
• grep -v → inverse : affiche les lignes qui NE correspondent PAS au motif
• ^# → lignes qui commencent par # (commentaires/directives)
• | wc -l → compte le nombre de lignes

Résultat : nombre de lignes actives (non commentaires) dans fichier.c
12
sed 's/18\([0-9][0-9]\)/19\1/g' transforme "1885" en :
  • A) 1985
  • B) 1895
  • C) 1819
  • D) Reste 1885
👁 Voir la correction

C'est l'exemple canonique du cours : transformer les dates 18XX en 19XX dans Les Misérables.

CH 03 ⚙️ Processus UNIX

Programme vs processus, états R/S/D/T/Z, fork, wait, exec, zombie, orphelin

15
Quelle est la différence fondamentale entre un programme et un processus ? 🔥 examen
  • A) Un programme s'exécute plus vite qu'un processus
  • B) Un programme est un fichier statique ; un processus est un programme en cours d'exécution avec des ressources
  • C) Un processus ne peut exécuter qu'un seul programme
  • D) Ils sont identiques, juste deux mots différents
👁 Voir la correction
✅ Bonne réponse : B
• Programme : fichier binaire statique sur le disque. Résultat d'une compilation. Objet inerte.
• Processus : programme en cours d'exécution. Dispose d'un état, d'un PID, de mémoire allouée, de descripteurs de fichiers, de registres.

Analogie : le programme = la recette de cuisine (statique) ; le processus = le cuisinier en train de cuisiner (dynamique).
16
Que retourne fork() dans le processus fils ? 🔥 examen
  • A) Le PID du père
  • B) Le PID du fils
  • C) 0
  • D) 1
👁 Voir la correction
✅ Bonne réponse : C — 0 dans le fils
Valeurs de retour de fork() — à savoir absolument par cœur :
• Dans le fils : 0
• Dans le père : PID du fils (valeur > 0)
• En cas d'erreur : -1 (pas de fils créé)

C'est la base du pattern if (pid == 0) { /* fils */ } else { /* père */ }
17
fork(); fork(); printf("Hello\n"); — combien de "Hello" ? 🔥 examen
  • A) 1
  • B) 2
  • C) 3
  • D) 4
👁 Voir la correction
✅ Bonne réponse : D — 4 "Hello"
Formule : n forks successifs → 2ⁿ processus au total
2 forks → 2² = 4 processus, chacun affiche "Hello" → 4 fois.

Arbre :
P0 ──fork1──→ P1
P0 ──fork2──→ P2
P1 ──fork2──→ P3
(4 processus, 4 "Hello")
18
if (fork()) fork(); printf("Hello\n"); — combien de "Hello" ? ⚠️ piège
  • A) 2
  • B) 3
  • C) 4
  • D) 6
👁 Voir la correction
✅ Bonne réponse : B — 3 "Hello"
• 1er fork() : crée P1. Dans P0 (père), retourne PID > 0 → vrai → P0 fait le 2ème fork.
• Dans P1 (fils), retourne 0 → faux → P1 n'exécute PAS le 2ème fork.
• 2ème fork() exécuté uniquement par P0 : crée P2.

3 processus : P0, P1, P2 → 3 "Hello"

⚠️ Erreur fréquente : penser que le fils fait aussi le 2ème fork. Seul le père exécute un if avec une condition vraie.
19
Un processus zombie est : 🔥 examen
  • A) Un processus bloqué en attente d'E/S disque
  • B) Un processus terminé dont le père n'a pas encore appelé wait()
  • C) Un processus suspendu par SIGSTOP
  • D) Un processus qui consomme 100% de CPU
👁 Voir la correction
✅ Bonne réponse : B
Un processus zombie (état Z) :
• A terminé son exécution
• Libère toutes ses ressources (mémoire, descripteurs…)
• Conserve son entrée dans la table des processus (pour que le père puisse récupérer son code de retour)
• Ne peut pas être tué (déjà mort !)
• Disparaît quand le père appelle wait()

Prévention : toujours appeler wait() ou waitpid() dans le père.
20
Peut-on tuer un processus zombie avec kill -9 PID ? ⚠️ piège
  • A) Oui, kill -9 tue tout
  • B) Non, le zombie est déjà terminé — impossible de tuer ce qui est mort
  • C) Oui, mais seulement en tant que root
  • D) Oui, après un délai de quelques secondes
👁 Voir la correction
✅ Bonne réponse : B
Un zombie est un processus déjà mort. Il ne reçoit plus de signaux, ne consomme pas de CPU, n'a plus de mémoire.

La seule façon de "nettoyer" un zombie : que son père appelle wait(). Si le père est lui-même mort, init (PID 1) adopte le zombie et le nettoie.
21
Ce code affiche-t-il "Fin" ?
char *args[] = {"ls", "-l", NULL}; execvp("ls", args); printf("Fin\n");
  • A) Oui, toujours, après l'exécution de ls
  • B) Non, jamais si execvp() réussit
  • C) Oui, mais seulement si ls retourne 0
  • D) Oui, dans le fils uniquement
👁 Voir la correction
✅ Bonne réponse : B — Non, jamais si execvp réussit
En cas de succès, execvp() remplace intégralement l'espace mémoire et le code du processus par celui du nouveau programme (ls). L'ancien code n'existe plus en mémoire.

Le printf("Fin\n") n'est atteint que si execvp() a échoué (retourne -1).

⚠️ Piège classique que le prof place exprès dans les examens.
22
Un processus orphelin est adopté par :
  • A) Le processus de PID 2
  • B) init (PID 1)
  • C) Le shell qui l'a lancé
  • D) Il devient zombie automatiquement
👁 Voir la correction
✅ Bonne réponse : B — init (PID 1)
Quand un père meurt avant son fils, ce fils devient orphelin. Le noyau le rattache automatiquement à init (PID 1), le premier processus du système.

Init appelle régulièrement wait(), ce qui évite que l'orphelin devienne zombie.
23
L'état D d'un processus (dans la sortie de ps) signifie : ⚠️ piège
  • A) Terminé (Dead)
  • B) En attente d'E/S disque, non interruptible
  • C) En cours d'exécution normale
  • D) Suspendu temporairement
👁 Voir la correction
✅ Bonne réponse : B
Tous les états à connaître :
ÉtatSignificationKill -9 ?
RRunning / prêt✅ Oui
SSleeping (attente interruptible)✅ Oui
DDisk sleep (E/S bloquante)❌ Non
TStopped (SIGSTOP)✅ Oui
ZZombie (terminé, non récolté)❌ Inutile
L'état D est le seul où kill -9 ne fonctionne pas car le processus est dans le noyau.
24
Quel signal ne peut être ni capturé ni ignoré par un processus ?
  • A) SIGTERM (15)
  • B) SIGINT (2)
  • C) SIGKILL (9)
  • D) SIGSEGV (11)
👁 Voir la correction
✅ Bonne réponse : C — SIGKILL (9)
• SIGKILL (9) : terminaison immédiate, ne peut être ni capturé, ni ignoré, ni bloqué.
• SIGSTOP (19) : suspension, même caractéristique (ne peut être ignoré).
• SIGTERM (15) : peut être capturé → permet au processus de se nettoyer.

Bonne pratique : utiliser kill PID (SIGTERM) d'abord, et kill -9 uniquement si le processus ne répond pas.
CH 04 🐚 Programmation Shell (Bash)

Variables, tests, boucles, redirections, pipes, scripts

25
VAR = valeur en bash (avec espaces) : 🔥 piège n°1
  • A) Définit correctement la variable VAR
  • B) Provoque une erreur de syntaxe
  • C) Exécute la commande VAR avec l'argument valeur
  • D) A le même effet que VAR=valeur
👁 Voir la correction
✅ Bonne réponse : B — Erreur de syntaxe
En bash, les espaces autour de = sont strictement interdits dans une affectation.

• VAR=valeur ✅ correct
• VAR = valeur ❌ bash interprète VAR comme une commande avec = et valeur comme arguments

Autres règles strictes du shell :
• Pas d'espace dans [ -f "$f" ] → [-f "$f"] ❌
• Guillemets " → substitution des variables ; apostrophes ' → littéral
26
Script appelé ./test.sh rouge bleu vert — que vaut $#, $1, $0 ?
  • A) $#=4, $1=test.sh, $0=rouge
  • B) $#=3, $1=rouge, $0=./test.sh
  • C) $#=3, $1=bleu, $0=rouge
  • D) $#=2, $1=rouge, $0=./test.sh
👁 Voir la correction
✅ Bonne réponse : B
Variables spéciales à savoir par cœur :
• $0 = nom du script → ./test.sh
• $1 = 1er argument → rouge
• $2 = 2ème argument → bleu
• $# = nombre d'arguments (sans $0) → 3

27
Que fait ce test : [ -f "$fichier" ] ? ⚠️ piège espaces
  • A) Vrai si $fichier est un répertoire

30
La ligne #!/bin/bash en tête d'un script s'appelle :
  • A) Un commentaire standard
  • B) Le shebang – indique l'interpréteur au noyau
  • C) Une déclaration de variable globale
  • D) Une directive de compilation

Le shebang (#! + chemin de l'interpréteur) est la toute première ligne d'un script exécutable. Il indique au noyau quel programme utiliser pour interpréter le fichier.

31
Sans export, une variable shell est-elle accessible dans un script fils ?
  • A) Oui, toutes les variables sont héritées automatiquement
  • B) Non, sans export la variable n'est accessible que dans le shell courant
  • C) Oui, uniquement pour les variables numériques

✅ Bonne réponse : B

CH 05 🧠 Allocation mémoire – Pile & Tas

Stack vs heap, malloc/free, fragmentation, pointeurs invalides

32
Quelle est l'erreur grave dans ce code ? 🔥 examen
int* f() { int n = 42; return &n; }
  • A) On ne peut pas retourner un pointeur en C
  • B) La variable n est sur la pile et libérée à la fin de f() → pointeur invalide (dangling pointer)
  • C) Il faut utiliser malloc() avant de retourner
  • D) Il manque un free()
👁 Voir la correction
✅ Bonne réponse : B
La variable n est une variable locale stockée sur la pile. À la fin de la fonction f(), son espace mémoire est libéré automatiquement.

Le pointeur retourné pointe vers une zone mémoire invalide → comportement indéfini (dangling pointer).

✅ Solution : allouer n sur le tas avec malloc() :
int *n = malloc(sizeof(int)); *n = 42; return n;
33
La fragmentation externe signifie :
  • A) L'espace alloué est plus grand que l'espace demandé
  • B) La mémoire libre est dispersée en petits blocs non contigus
  • C) La pile est pleine
  • D) malloc() retourne NULL
👁 Voir la correction
✅ Bonne réponse : B
• Fragmentation externe : mémoire totale libre suffisante MAIS répartie en petits blocs éparpillés → impossible d'allouer un gros bloc contigu.
• Fragmentation interne : bloc alloué plus grand que demandé → gaspillage.

Exemple : 120 Mo libres mais on ne peut pas allouer 100 Mo car les 120 Mo sont en 3 blocs de 40 Mo non adjacents.
34
Donnez la syntaxe pour allouer un tableau de 10 entiers et le libérer : 🔥 examen
  • A) int tab[10]; free(tab);
  • B) int *tab = malloc(10); free(tab);
  • C) int *tab = malloc(10 * sizeof(int)); free(tab);
  • D) int *tab = new int[10]; delete[] tab;
👁 Voir la correction
✅ Bonne réponse : C
• malloc(10) ❌ : alloue 10 octets, pas 10 entiers
• malloc(10 * sizeof(int)) ✅ : alloue 10 × taille d'un int (généralement 40 octets)
• Toujours vérifier : if (tab == NULL) { perror("malloc"); exit(1); }
• delete[] → syntaxe C++, pas C

Règle d'or : malloc = utiliser sizeof() !
35
Tableau comparatif Pile vs Tas — quel point est INCORRECT ? ⚠️ piège
  • A) La pile est gérée automatiquement par le compilateur
  • B) Le tas ne peut pas engendrer de fuite mémoire
  • C) La pile fonctionne en LIFO
  • D) La fragmentation est possible sur le tas, pas sur la pile
👁 Voir la correction
✅ Bonne réponse : B — C'est FAUX
B est faux : le tas peut engendrer des fuites mémoire (si on oublie free()).

Les 3 autres affirmations sont vraies :

Pourquoi globale++ sans mutex peut produire un résultat incorrect avec 2 threads ? 🔥 examen
  • A) C'est une opération interdite sur les variables globales
  • B) Elle est compilée en 3 instructions (LOAD/ADD/STORE) — l'ordonnanceur peut interrompre entre elles
  • C) Les variables globales ne peuvent pas être modifiées par plusieurs threads
  • D) C'est trop lent pour les threads
👁 Voir la correction
✅ Bonne réponse : B
globale++ n'est pas atomique — elle se décompose en :
LOAD  R1 ← [globale]   // 1. Lire
ADD   R1 = R1 + 1       // 2. Incrémenter
STORE [globale] ← R1    // 3. Écrire
Si l'ordonnanceur interrompt le Thread A entre LOAD et STORE, le Thread B lit l'ancienne valeur, fait son +1, puis A écrase avec son +1 → perte d'une incrémentation.

Solution : encadrer de pthread_mutex_lock / pthread_mutex_unlock.

Deux threads incrémentent chacun 10 000 fois une variable globale = 0. La valeur finale sera-t-elle toujours 20 000 sans mutex ?

  • B) Non, la valeur finale sera souvent inférieure à 20 000 (race condition)
  • C) Oui, le compilateur optimise cette opération
  • D) Non, la valeur sera exactement 10 000
👁 Voir la correction
✅ Bonne réponse : B
Sans mutex, des incrémentations sont perdues à cause de la race condition. La valeur finale est non déterministe et généralement inférieure à 20 000 (souvent autour de 19 000+).

Avec le mutex :

pthread_mutex_unlock(&mutex);
→ Résultat garanti : 20 000.
